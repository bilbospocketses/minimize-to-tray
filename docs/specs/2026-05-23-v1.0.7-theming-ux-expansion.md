# v1.0.7 Theming + UX Expansion — Design Specification

**Date:** 2026-05-23
**Target release:** v1.0.7 (in-flight)
**Status:** Approved — mid-implementation amendment to v1.0.7
**Supplements:** [`2026-05-23-v1.0.7-rescue-mode-design.md`](./2026-05-23-v1.0.7-rescue-mode-design.md)

## Background

The original v1.0.7 spec described a theme-aware rescue dialog reusing the same conventions as the About dialog. After landing Tasks 1-11, manual smoke testing surfaced two real-world issues:

1. **Dark-mode theming is incomplete.** The native Windows-painted controls (Button, ListView header, Checkbox) do not honor AHK's `Opt` color hints — they render in the system default theme, which is light. In a dark dialog this looks broken: white buttons, white ListView header, intro Text in black on a dark background. The About dialog has the same latent bug — its OK button and Run-on-login checkbox were never correctly themed in dark mode either, just unnoticed until now.

2. **Restore Selected UX leaves orphans.** Unchecked rows were left in `hidden.json` to re-prompt next launch, but the windows themselves stayed orphaned on the desktop (hidden, unreachable). Users naturally expect "unchecked" to mean "leave accessible via the tray," not "abandon until next crash."

This amendment expands v1.0.7 with the smallest correct fix for each.

## A. UX redesign — exit semantics

### New behavior matrix

The rescue dialog has three buttons, plus implicit Esc / Close-X. After the redesign, **no exit path leaves orphans on the desktop** — every hidden window is either visible or accessible via the tray after the dialog closes.

| Action | Checked rows | Unchecked rows | `hidden.json` after |
|---|---|---|---|
| **Restore Selected** | `WinShow` + `HiddenState_Remove(hwnd)` | re-register into `Groups` tray system; entry stays in file | only unchecked rows remain |
| **Restore All** | `WinShow` all + `HiddenState_Remove` each | n/a (no unchecked rows by definition) | cleared (matches empty `Groups`) |
| **Send All to Tray** (was: Cancel) | re-register all into `Groups` tray; entries stay in file | re-register all into `Groups` tray; entries stay in file | unchanged (still has all N entries) |
| **Esc / Close X** | same as Send All to Tray | same | same |

Invariant: `hidden.json` always mirrors in-memory `Groups` state. Every entry in the file corresponds to a window the running app is tracking.

### Button label changes

- "Cancel" → "Send All to &Tray". The new label matches the action. Default button remains "Restore All" (Enter-to-default).

### Intro text reword

Replace:

> Found N windows hidden by a previous session of minimize-to-tray.
> Uncheck any you want to leave hidden.

With:

> Found N windows hidden by a previous session of minimize-to-tray.
> Restore the checked rows to view; unchecked rows return to the tray.

### Required code shape

New helper:

```ahk
StashAlreadyHidden(hwnd, pid, procName, procPath, title)
```

Same Groups + tray-icon registration logic as `HideAndStash`, but skips the `WinHide` call (window is already hidden from the previous session) AND skips the `HiddenState_Append` call (entry is already in the file from the previous session — preserved as-is, original `hiddenAt` retained).

To avoid duplicating ~30 lines of Groups + ShellNotifyAdd logic between `HideAndStash` and `StashAlreadyHidden`, factor out a shared private helper:

```ahk
RegisterTrayGroup(hwnd, pid, procName, procPath)
```

Both `HideAndStash` and `StashAlreadyHidden` call it. `HideAndStash` then calls `WinHide` + `HiddenState_Append`; `StashAlreadyHidden` skips both.

The three OnRescue* handlers rewrite to follow the matrix above.

## C. GDI+ owner-draw scope

Five places where native Win32 rendering shows through:

1. **Rescue dialog buttons** (Restore Selected, Restore All, Send All to Tray)
2. **About dialog OK button**
3. **About dialog Run-on-login Checkbox**
4. **Rescue ListView header row**
5. **Rescue intro Text color** — trivial; not owner-draw, just `txt.Opt("c" pal.text)` after `AddText`.

### Approach: BS_OWNERDRAW + WM_DRAWITEM for buttons and checkbox

The cleanest AHK 2 path for buttons + checkbox:

- Apply `+0x0B` to the control style (`BS_OWNERDRAW = 0x0B`) — disables native rendering, posts `WM_DRAWITEM` to the parent for each paint.
- Hook `WM_DRAWITEM` on the parent Gui via `OnMessage(0x002B, OnGuiDrawItem)`.
- `OnGuiDrawItem(wParam, lParam, msg, hwnd)` reads the `DRAWITEMSTRUCT` from `lParam` and dispatches by control hwnd:
  - If hwnd matches one of the rescue dialog's button hwnds → `DrawOwnerButton(dis, themePal, isDefault, isDefaultStyle)`
  - If hwnd matches About OK button → same `DrawOwnerButton`
  - If hwnd matches About Run-on-login checkbox → `DrawOwnerCheckbox(dis, themePal, isChecked)`

### Approach: NM_CUSTOMDRAW for ListView header

`WM_NOTIFY` from the ListView with `NM_CUSTOMDRAW` notification code. Hook via `LV.OnEvent("Notify", ...)` if AHK 2 supports it; otherwise subclass the header via `SetWindowSubclass` and handle `WM_NOTIFY` directly. Custom-paint each header item per `pal.headerBg` + `pal.headerFg` and let Windows handle the rest.

If implementing NM_CUSTOMDRAW proves too brittle in AHK 2, the fallback is to hide the native header (`-Hdr` style on ListView), add a Text-control header row above the ListView, and manually re-layout on column resize. We treat this as the fallback approach, not the primary.

### Required helper code shapes

```ahk
DrawOwnerButton(dis, pal, isDefault) {
    ; dis = DRAWITEMSTRUCT pointer.
    ; Reads CtlID, hwndItem, itemState (selected/focused/hot), rcItem.
    ; Picks bg/fg/border colors based on itemState + isDefault.
    ; Fills with bg, draws border, draws text centered via DrawTextW, draws focus ring if focused.
}

DrawOwnerCheckbox(dis, pal, isChecked) {
    ; Same DRAWITEMSTRUCT pattern; draws box on left, glyph if checked, label on right.
}

OnGuiDrawItem(wParam, lParam, msg, hwnd) {
    ; Global WM_DRAWITEM dispatcher. Reads DRAWITEMSTRUCT.hwndItem, routes to the right
    ; per-control draw helper. Returns TRUE so default paint is suppressed.
}
```

State detection in `dis.itemState` (bit flags):
- `ODS_SELECTED = 0x0001` — pressed
- `ODS_FOCUS = 0x0010` — keyboard-focused
- `ODS_HOTLIGHT = 0x0040` — mouse-hovered (only fires when the button window has the `TME_HOVER` track — which `BS_OWNERDRAW` buttons do by default when the user mouses over them, but only AFTER `TrackMouseEvent` is registered)
- `ODS_DEFAULT = 0x0020` — default pushbutton

`BS_OWNERDRAW` buttons do NOT auto-track mouse hover. To get `ODS_HOTLIGHT` correctly the parent dialog needs to call `TrackMouseEvent` on each owner-drawn button as the mouse enters its rect. This is fiddly. **Pragmatic compromise:** skip hover state for v1.0.7. Buttons render normal/pressed/focused/default only. Add hover in a future polish pass if it bothers anyone visually.

### Hi-DPI

The existing app is hi-DPI aware via Windows' default per-monitor V2 awareness for AHK 2. All GDI+ measurements are in logical pixels. Use `A_ScreenDPI` if any pixel-perfect drawing needs scaling, but most paint operations (`FillRect`, `DrawTextW`) auto-scale via the DC.

### Out of scope (v1.0.7)

- Hover state on owner-drawn buttons (requires `TrackMouseEvent` plumbing — defer)
- Animated state transitions (defer)
- Per-control accent customization beyond the palette
- Theming the tray-icon group menu / per-app tray flyouts (Windows-native, already system-themed)

## Open questions

None — all design decisions resolved in the 2026-05-23 mid-implementation brainstorm:

- Bundle everything into v1.0.7 (rather than split to v1.0.7.1).
- "Send All to Tray" replaces "Cancel" entirely (no separate Cancel button).
- `hidden.json` mirrors `Groups`: not blindly cleared on dialog exit.
- Skip hover state on owner-drawn buttons for v1.0.7.
- NM_CUSTOMDRAW is the primary path for the ListView header; hidden-header fallback is acceptable if AHK 2 plumbing proves brittle.
