# minimize-to-tray — Design Specification

**Date:** 2026-05-22
**Target release:** v1.0.0
**Status:** Approved (brainstorm) — pending writing-plans → implementation

## Project

A small AutoHotkey v2 utility that adds "minimize the current window to the system tray" to Windows, with windows of the same process grouped under a single tray icon.

## Motivation

Windows does not ship with built-in minimize-to-tray. The existing utilities in this space (RBTray, MinTrayR, TrayIt, various AHK community scripts) are stale, fork-fragmented, or behave inconsistently on modern Win32. This project is a from-scratch v1 with a deliberately tiny surface area aimed at consistent behavior across modern desktop apps.

## Scope

### In scope (v1.0.0)

- Ships as a **Velopack-packaged Windows installer** (`MinimizeToTraySetup-1.0.0.exe`). User downloads Setup, runs it, app installs to a Velopack-managed location with proper uninstaller and Update.exe deployed alongside. Single self-contained Ahk2Exe-compiled .exe (no AutoHotkey runtime required on the user's machine) is the inner application bundled into the Velopack package.
- **Update awareness**: a small companion `updater-helper.exe` (.NET 9 self-contained, ~6-10 MB) uses `Velopack.Sdk.UpdateManager` to check GitHub Releases for newer versions. AHK script invokes it via `WScript.Shell.Exec` on startup; if a newer version is reported, the About menu surfaces a **pulsing blue dot** with hover tooltip ("Update available: vX.Y.Z") and click action ("download + apply + restart" via the helper's `update` command).
- An **always-visible "app" tray icon** while the program is running, distinct from the per-group icons. Provides:
  - Hover tooltip serving as the trigger reminder: app name plus the two shortcuts (`Win+Shift+Z` and "middle-click title bar").
  - Right-click menu with "About" and "Exit".
  - No left-click action (single-click on the app icon does nothing — left-click is reserved for the per-group icons described below).
- Two triggers — both built into the program, no configuration:
  - `Win+Shift+Z` to minimize the focused window to the tray.
  - Middle-click on a window's title bar to minimize that window.
- Group minimized windows by **process executable name** (e.g., `notepad.exe`). One tray icon per group, in addition to the always-visible app icon.
- Per-group tray icon visual: extracted from the process's own .exe via `ExtractIconEx`.
- Per-group tooltip on hover: `<ProcessName> (<N> windows)`.
- Left-click on a per-group tray icon: restore the most recently minimized window in that group (LIFO).
- Right-click on a per-group tray icon: menu containing
  - Each minimized window listed by its title (selecting restores that specific window — out-of-order pick).
  - Separator.
  - "Restore All".
  - "Close All".
- Orphan cleanup: event-driven via `SetWinEventHook(EVENT_OBJECT_DESTROY)` — per-group tray icon updates immediately when a tracked window dies externally.
- OnExit handler: every hidden window is restored before the program quits.
- No GUI windows. Everything lives in the tray.
- Public, MIT-licensed, OSS on GitHub.

### Out of scope (v1.0.0)

Parked for "if it gains traction" follow-up work:

- Configurable hotkey or alternate triggers.
- Per-app grouping overrides (e.g., split Chrome by profile, merge Office apps).
- Persistent state across script restart / reboot.
- Auto-start with Windows (user wires via Task Scheduler / Startup folder).
- App icon + count badge composite.
- Settings UI / settings file.
- Crash-recovery rescue mode (see Recovery section below).
- Localization.

## Architecture

Single AutoHotkey v2 script. Single process. All Win32 API access via `DllCall` — no external libraries.

### State

```
Groups : Map(processName → GroupRecord)

GroupRecord := {
  windows : Array of Integer,   ; hwnds, LIFO stack — push on minimize, pop on click
  trayUid : Integer,            ; stable Shell_NotifyIcon UID
  hIcon   : Integer             ; HICON handle (freed on group destroy)
}
```

Reverse-lookup helper `HwndToGroup(hwnd) → processName` walks `Groups` linearly (N is small — typically <10 groups in practice).

UID allocator: a script-scope counter starting at 1, incremented per `NIM_ADD`. UIDs are never reused within one script run — avoids tray race conditions where a stale callback hits a newly-allocated icon.

The script owns a hidden stub GUI window whose hwnd is used as the recipient for `Shell_NotifyIcon`'s callback messages. Created at script init. This hwnd is NEVER shown to the user — its sole purpose is to receive WM_USER+1 messages from the tray icons.

### Always-visible app tray icon

Uses AHK v2's built-in `A_TrayMenu` and tray-icon facilities (which are separate from the `Shell_NotifyIcon` icons we manage manually for groups — both can coexist as distinct entries in the Windows notification area).

At script init:

```ahk
TraySetIcon(A_ScriptDir "\assets\app.ico")   ; or the embedded icon when running as compiled .exe
A_IconTip := "minimize-to-tray`n"
          .  "Win+Shift+Z: minimize focused window`n"
          .  "Middle-click title bar: minimize that window"

A_TrayMenu.Delete()                          ; remove all of AHK's default items
A_TrayMenu.Add("&About", ShowAbout)
A_TrayMenu.Add()                             ; separator
A_TrayMenu.Add("E&xit", (*) => ExitApp())
A_TrayMenu.Default := "&About"               ; double-click does nothing harmful
```

`ShowAbout` shows a small `MsgBox` with: app name, version, GitHub URL, the two shortcut strings. No clickable hyperlink in v1 (a plain MsgBox is fine).

The total tooltip string fits well within Windows' 128-wide-char `szTip` limit (~90 chars at current wording).

### Triggers

```ahk
#+z::MinimizeFocused()                          ; Win+Shift+Z

#HotIf MouseOverTitleBar()
MButton::MinimizeUnderCursor()                  ; middle-click consumed, not passed through
#HotIf
```

**`MouseOverTitleBar()`** predicate:

1. `MouseGetPos &x, &y, &hwnd` — `x,y` in screen coords.
2. `SendMessage WM_NCHITTEST (0x84), 0, ((y << 16) | (x & 0xFFFF)), , "ahk_id " hwnd` — packed screen coords in lParam.
3. Return `result == HTCAPTION (2)`.

For traditional title-bar apps (Notepad, File Explorer, classic Win32) this is bulletproof. For apps with custom-drawn title bars (Chrome, Edge, Office, VS Code) only the drag-handle / caption region reports HTCAPTION — middle-click on, e.g., a Chrome tab strip will not trigger us (the tab consumes the click instead). This is documented as a known limitation in the README.

### Minimize flow

`HideAndStash(hwnd)`:

1. Resolve `procName` via `ProcessGetName(hwnd)` and `exePath` via `ProcessGetPath(hwnd)`.
2. If `Groups.Has(procName)`:
   - Push `hwnd` onto `Groups[procName].windows`.
3. Else (first window of this type):
   - `ExtractIconEx(exePath, 0, &hLarge, 0, 1)` → `hIcon`. On failure, fall back to the script's own default icon.
   - Allocate `trayUid`.
   - Build `NOTIFYICONDATA`: `hWnd = scriptGuiHwnd`, `uID = trayUid`, `uFlags = NIF_ICON | NIF_MESSAGE | NIF_TIP`, `uCallbackMessage = WM_USER + 1 (0x0401)`, `hIcon = hIcon`, `szTip = procName`.
   - `Shell_NotifyIcon(NIM_ADD, &nid)`.
   - `Groups[procName] := { windows: [hwnd], trayUid: trayUid, hIcon: hIcon }`.
4. `WinHide("ahk_id " hwnd)`.
5. Rebuild `NOTIFYICONDATA` with `szTip = procName " (" Groups[procName].windows.Length " windows)"`. `Shell_NotifyIcon(NIM_MODIFY, &nid)`.

### Tray callback (WM_USER+1)

Registered via `OnMessage(0x0401, OnTrayMessage)`. The hidden stub GUI is the recipient because `Shell_NotifyIcon` requires an hwnd.

The callback receives `wParam = tray UID`, `lParam = mouse-event code`.

Dispatch:

- `WM_LBUTTONUP (0x0202)`: look up group by UID → pop top of `.windows` → `WinShow + WinActivate`. If stack now empty → destroy group (`Shell_NotifyIcon(NIM_DELETE)`, `DestroyIcon(hIcon)`, `Groups.Delete(procName)`).
- `WM_RBUTTONUP (0x0205)` or `WM_CONTEXTMENU (0x007B)`: build the right-click menu (see below) and `Menu.Show()` at the cursor.
- All other mouse events: ignore.

### Right-click menu

```
menu := Menu()
For each hwnd in group.windows (top of stack first):
  title := WinGetTitle("ahk_id " hwnd)
  if title = ""
    title := "(no title)"
  menu.Add(title, RestoreSpecific.Bind(group, hwnd))
menu.Add()                                  ; separator
menu.Add("Restore &All", RestoreAll.Bind(group))
menu.Add("Close A&ll",   CloseAll.Bind(group))
menu.Show()
```

- `RestoreSpecific(group, hwnd)`: remove `hwnd` from `group.windows`, `WinShow + WinActivate hwnd`. If group empty → destroy group.
- `RestoreAll(group)`: for each hwnd in `group.windows` (top first), `WinShow + WinActivate`. Destroy group.
- `CloseAll(group)`: for each hwnd, `WinClose` (NOT `WinKill` — gives apps a chance to prompt about unsaved changes). The `EVENT_OBJECT_DESTROY` hook handles bookkeeping as each window dies. If a `WinClose` is blocked (e.g., the user cancels an unsaved-changes prompt), that window remains alive in our stack — the tray icon persists with the surviving windows and an updated count.

Titles are sampled at menu-build time. If a title changes between menu show and click, the displayed label is stale but the hwnd binding is still correct.

### Orphan cleanup (event-driven)

At script init:

```ahk
hookProc := CallbackCreate(OnWinEvent, "Fast")
hHook := DllCall("SetWinEventHook"
                 , "UInt", 0x8001    ; EVENT_OBJECT_DESTROY
                 , "UInt", 0x8001
                 , "Ptr",  0
                 , "Ptr",  hookProc
                 , "UInt", 0         ; idProcess (0 = all processes)
                 , "UInt", 0         ; idThread
                 , "UInt", 0         ; WINEVENT_OUTOFCONTEXT
                 , "Ptr")
```

`OnWinEvent(hook, event, hwnd, idObject, idChild, eventThread, eventTime)`:

1. Filter: bail unless `idObject == OBJID_WINDOW (0)` and `idChild == CHILDID_SELF (0)` — `EVENT_OBJECT_DESTROY` fires for child objects too.
2. Reverse-lookup `hwnd → procName` via `HwndToGroup`.
3. If found:
   - Remove `hwnd` from the group's `.windows` array.
   - If now empty → destroy the group.
   - Else → refresh the group's tooltip via `Shell_NotifyIcon(NIM_MODIFY)`.

### OnExit handler

```ahk
OnExit(Cleanup)

Cleanup(reason, code) {
  for procName, group in Groups {
    for hwnd in group.windows
      try WinShow("ahk_id " hwnd)   ; window may have died between hook firing and exit
    DllCall("DestroyIcon", "Ptr", group.hIcon)
    Shell_NotifyIcon(NIM_DELETE, group.trayUid)
  }
}
```

Handles graceful exits (script reload, manual quit from AHK's default tray). Hard crashes (process killed) bypass this — see Recovery.

### Recovery

If the script crashes hard with windows still hidden, those windows remain hidden in their owning processes and there is no in-app way to recover them in v1. Mitigations:

- The script is small; hard crashes should be rare.
- Manual recovery options for the user:
  - Restart the owning app (closes the hidden window with the rest of the process).
  - Kill via Task Manager (apps with auto-save survive).

Documented in the README's "Known limitations". A `--rescue` mode that enumerates hidden top-level windows and offers to restore them is parked for post-v1.

## Failure modes & edge cases

| Case | Behavior |
| --- | --- |
| `ExtractIconEx` returns no icon (rare — system processes) | Fall back to the script's default icon. |
| Hotkey fires with no foreground window (desktop focused) | `WinGetID("A")` returns 0 → no-op. |
| Middle-click on window with no caption (full-screen game, splash) | Predicate returns false → click passes through. |
| Middle-click on a Chrome tab / VS Code tab strip / custom title region | Predicate returns false for the non-caption region → click passes through. Documented limitation. |
| User restores a window via our tray, then minimizes it normally (Win+Down), then re-triggers our hotkey | Treated as a fresh minimize. Standard minimize state is overridden by `WinHide`. |
| User triggers minimize on a window already in our stack | Cannot happen — `WinHide` removes the window from Alt+Tab and the taskbar. |
| Right-click menu shown, user clicks outside | Menu closes. No-op. |
| `Shell_NotifyIcon(NIM_ADD)` fails (rare — usually only post-explorer.exe-crash) | Retry once after 500ms. If still fails, abort the minimize (do NOT call `WinHide`) and show a `MsgBox` to the user. |
| Window changes title between menu-build and click | Menu label is stale, hwnd binding is still correct → restore works fine. |
| First-run SmartScreen warning on the unsigned .exe | User sees "Windows protected your PC". Documented in README: click "More info" → "Run anyway". Resolved post-traction by code signing. |
| User has Windows 11 collapsed-tray-icons mode and the per-group icons hide in the overflow flyout | Functions identically — tooltips and click handling work whether icons are pinned or in the flyout. Document in README so users know to pin if they want the icons always visible. |

## Testing approach

Manual smoke walkthrough at first commit and again before tagging v1.0.0:

1. Open three Notepad windows, type unique text in each so titles differ.
2. `Win+Shift+Z` on each → one Notepad tray icon, tooltip "Notepad (3 windows)".
3. Left-click tray icon → most recent Notepad restores. Repeat twice → tray icon disappears after the third click.
4. Repeat with two Notepad + two Calculator windows → two distinct tray icons, side by side, each with its own app icon.
5. Right-click the Notepad icon → menu lists all three titles (top of stack first) + Restore All + Close All. Pick the bottom title → it restores out of order.
6. Right-click → "Restore All" → all three restore, icon vanishes.
7. Right-click → "Close All" → all three close, icon vanishes.
8. Middle-click on a Notepad title bar → minimizes. Middle-click on a Chrome tab strip → does NOT minimize (passes through to Chrome).
9. Minimize a Notepad to tray, kill `notepad.exe` via Task Manager → tray icon's count decreases within ~100ms (hook fires); if it was the only window, the icon disappears.
10. Minimize a window, then exit the script via AHK's default tray "Exit" → window restores before the script quits.

No automated test suite for v1. AHK has no standard test framework that justifies the setup cost for a single ~400-line script.

## EXE packaging & build

Compiled with **Ahk2Exe**, which ships with the AHK v2 install at `C:\Program Files\AutoHotkey\Compiler\Ahk2Exe.exe`. Output: a self-contained ~1–2 MB Windows .exe that bundles the AHK runtime.

### Build invocation

```powershell
Ahk2Exe.exe `
  /in   "minimize-to-tray.ahk" `
  /out  "dist\minimize-to-tray.exe" `
  /icon "assets\app.ico" `
  /base "<v2-AutoHotkey64.exe>"
```

The base is the 64-bit AHK v2 runtime stub. `/icon` embeds the .ico in the exe's PE resources so Explorer/taskbar/Alt+Tab all show the right icon.

A `build.ps1` wrapper at the repo root automates this:
- Resolves Ahk2Exe via the standard install path (no system PATH dependency).
- Cleans `dist/` and rebuilds.
- Verifies the output exists and prints its size.

### Asset: `assets/app.ico`

A real custom icon is created during implementation (a small purpose-built design — probably a downward arrow into a tray glyph). Until that's finalized, a placeholder generated via ImageMagick from a simple SVG.

### Distribution

- The .exe is the primary release asset.
- The .ahk source is also published in the release for users who already have AHK installed and prefer to run/inspect the source.
- The compiled .exe is unsigned in v1.0.0 — Windows SmartScreen will warn on first run. README documents the "More info → Run anyway" workaround. Code signing is a post-traction follow-up.

### CI

No CI in v1.0.0. Local build via `build.ps1` and manual `gh release create` upload. A GitHub Actions workflow that builds + uploads on tag push is parked for the post-traction follow-up.
