# v1.0.7 Crash-Recovery Rescue Mode + v1.0.8 Diagnostic Hotkey — Design Specification

**Date:** 2026-05-23
**Target release:** v1.0.7
**Status:** Approved (brainstorm) — pending writing-plans → implementation

## Project

Add crash-recovery rescue mode: if the app crashed (or exited uncleanly) while one or more user windows were hidden via the tray, surface a modal dialog on the next launch that offers to restore them. Track hidden windows in a JSON state file so we can validate and prune before offering anything.

Also ship a small always-on diagnostic hotkey (`Ctrl+Alt+Win+Z`) that dumps active-window properties to the clipboard. This is the v1.0.8 prep step: a known bug exists in the Electron / Chromium app class (canonical test case: VS Code; other suspects: Discord, Slack, Teams, GitHub Desktop, Spotify, the new Outlook, Edge, Chrome). These apps accept the minimize hotkey, the tray icon appears, but the window does not visually hide — likely because the Chromium compositor + Win11 system-backdrop machinery maintains the visible surface independent of the `SW_HIDE` window state. We need real captured properties of the misbehaving windows before designing the fix.

## Motivation

### Rescue mode

The tray-icon group system in v1.0.0+ is the only path to un-hide a window. If the AHK process dies (crash, force-kill via Task Manager, abnormal shutdown), the tray icon goes with it — and every window currently `WinHide`'d is orphaned. The user can't see those windows, can't find them on the taskbar (they don't show because they're hidden), and can't restore them without killing and relaunching the owning app. From a user-trust standpoint this is the worst possible failure mode the app can have. Rescue mode closes that hole.

### Diagnostic hotkey

The user has reported that some apps — confirmed with VS Code (`Code.exe`) as a canonical test case — accept the `Win+Shift+Z` minimize hotkey but do not actually hide. The tray icon appears (so `HideAndStash` runs successfully), but the rendered window surface stays painted on the desktop. The bug class is Electron / Chromium-based apps with Win11 system backdrops; other likely-affected apps include Discord, Slack, Teams, GitHub Desktop, Spotify, the new Outlook client, and the Chromium-based browsers themselves (Edge, Chrome, Brave). Likely cause: the Chromium compositor + DWM backdrop machinery maintains a visible surface that ignores `ShowWindow(SW_HIDE)`. Before designing the v1.0.8 fix, we need empirical evidence — class, styles, ancestor chain, DWM cloak state, system-backdrop attribute. A small always-on diagnostic hotkey captures all of this in one keypress, usable against any active window.

## Scope

### In scope (v1.0.7)

**Rescue mode:**

- Persistent bookkeeping at `%LOCALAPPDATA%\bilbospocketses\minimize-to-tray\hidden.json`.
- One entry per currently-hidden window: hwnd, pid, procName, procPath, title, hiddenAt timestamp.
- Append on every successful `HideAndStash`. Remove on every successful `RestoreSpecific` / `RestoreAllInGroup`. Atomic rewrite via temp-file + `MoveFileEx(MOVEFILE_REPLACE_EXISTING)`.
- Startup discovery: validate every entry (window still exists, same process, currently hidden); silently prune stale entries; survivors only.
- Modal rescue dialog (theme-aware, matches About-dialog styling):
  - ListView with columns: checkbox / process name / window title / hidden-at time
  - All rows checked by default
  - Buttons: **Restore Selected**, **Restore All**, **Cancel**
  - Cancel / X close → clear bookkeeping silently; orphaned windows stay hidden on the desktop (explicit user opt-out)
- Restored windows return as normal, taskbar-tracked windows — they are NOT re-registered in the `Groups` map / tray-icon system.
- Best-effort file I/O: rescue-side failures log to `%LOCALAPPDATA%\bilbospocketses\minimize-to-tray\rescue.log` but never block the user's hide / restore actions.

**Diagnostic hotkey (v1.0.7 prep for v1.0.8 fix work):**

- New hotkey `Ctrl+Alt+Win+Z` (in addition to existing `Win+Shift+Z`).
- Captures active-window properties, formats as multi-line text, copies to clipboard.
- Shows a small toast / non-blocking notification: "Window dump copied to clipboard."
- Always on (not gated behind a CLI flag) — small enough to leave in production. v1.0.8 may keep, remove, or repurpose.

**Uninstall cleanup:**

- Extend the existing `--veloapp-uninstall` handler to also remove `%LOCALAPPDATA%\bilbospocketses\minimize-to-tray\` (the new rescue state directory) in addition to the HKCU registry wipe that v1.0.2 already performs. Idempotent: `DirDelete(APP_DATA_DIR, true)` inside a try/catch — silently swallow if the folder doesn't exist.

### Out of scope (v1.0.7)

- The actual fix for the Electron / Chromium minimize bug. Moves to v1.0.8 after diagnostic output is captured and analyzed.
- Auto-restore-on-startup without a prompt (deliberately rejected during brainstorm — modal dialog preserves user agency).
- Tray-balloon prompt instead of modal (rejected — Win10/11 balloon-click reliability is poor).
- Manual rescue invocation via CLI flag or tray-menu entry (rejected — in-session needs are already covered by the existing per-app tray-icon-group restore path; auto-on-startup handles the crash-recovery case).
- Broad system scan of all hidden windows on the desktop (rejected — false positives from apps that legitimately hide themselves to tray, e.g. Discord, Spotify, Steam).
- Persisting hidden-window state in the registry (rejected — list-of-windows shape doesn't fit `REG_SZ` / multi-value layout well; HKCU keeps scalar config only).
- A configurable "auto-restore everything on clean exit" feature (out of scope — separate idea, would compete with the explicit hide-to-tray UX).

## Architecture

Single-file edit to `minimize-to-tray.ahk`. No new dependencies, no Velopack changes, no GitHub workflow changes. Companion `updater-helper.exe` is unaffected.

### State

```ahk
global APP_DATA_DIR        := ""                          ; resolved to %LOCALAPPDATA%\bilbospocketses\minimize-to-tray
global HIDDEN_STATE_FILE   := ""                          ; APP_DATA_DIR "\hidden.json"
global HIDDEN_STATE_TMP    := ""                          ; APP_DATA_DIR "\hidden.json.tmp"
global RESCUE_LOG_FILE     := ""                          ; APP_DATA_DIR "\rescue.log"
global rescueGui           := 0                           ; modal Gui object while open, 0 otherwise
```

Resolved in `Initialize()` via `EnvGet("LOCALAPPDATA")` join `"\bilbospocketses\minimize-to-tray"`; directory created with `DirCreate(APP_DATA_DIR)` (idempotent — no error if it already exists).

### Hidden state file format

```json
{
  "version": 1,
  "windows": [
    {
      "hwnd": 1968412,
      "pid": 13456,
      "procName": "Code.exe",
      "procPath": "C:\\Users\\jscha\\AppData\\Local\\Programs\\Microsoft VS Code\\Code.exe",
      "title": "minimize-to-tray.ahk - minimize-to-tray - Visual Studio Code",
      "hiddenAt": "2026-05-23T14:32:15Z"
    }
  ]
}
```

- `version: 1` reserved for future schema migrations.
- ISO-8601 UTC timestamps for `hiddenAt` (formatted as `FormatTime(A_NowUTC, "yyyy-MM-ddTHH:mm:ssZ")`).
- `procPath` is the full path returned by `ProcessGetPath(pid)`. Used to detect PID recycling (rare, but possible if the system has high process churn between AHK sessions).
- JSON is minified on write (no pretty-printing) — file is internal state, not human-edited.

### Hidden-state module (inlined functions in `minimize-to-tray.ahk`)

```ahk
HiddenState_Read()                          ; returns Array of entries, or [] on missing/parse-error
HiddenState_Append(hwnd, pid, procName, procPath, title)
HiddenState_Remove(hwnd)
HiddenState_RewriteSurvivors(survivors)     ; called by RescueOrphanedWindows after pruning
HiddenState_Clear()                         ; truncate to { version: 1, windows: [] }
HiddenState_AtomicWrite(jsonText)           ; writes HIDDEN_STATE_TMP, MoveFileEx to HIDDEN_STATE_FILE
```

All write functions use atomic rename via the Win32 `MoveFileExW(MOVEFILE_REPLACE_EXISTING=1)`. A crash mid-write leaves either the old file or the new file intact, never a partial.

JSON parsing: AHK 2.0 has no built-in JSON. Two options:
1. Hand-rolled parser (~80 lines, sufficient for our flat schema).
2. Vendor `JSON.ahk` from cocobelgica/AutoHotkey-JSON or similar (~1 file, MIT-licensed).

**Decision:** hand-rolled. Our schema is rigid (array of flat objects), we never decode arbitrary external JSON, and avoiding a vendored dep keeps the script self-contained. Hand-roll covers `{`, `}`, `[`, `]`, `"`, `,`, `:`, escapes (`\\`, `\"`, `\n`, `\r`, `\t`, `\u####`), numbers, `true`/`false`/`null`. ~100 lines including encoder.

### Integration points in existing flow

**`HideAndStash(hwnd)` — line 800 area:**

After the existing `WinHide("ahk_id " hwnd)` (line 842), add:

```ahk
try {
    title := WinGetTitle("ahk_id " hwnd)
    HiddenState_Append(hwnd, pid, procName, exePath, title)
} catch as e {
    LogRescue("HiddenState_Append failed for hwnd=" hwnd ": " e.Message)
}
```

`pid`, `procName`, `exePath` are already in scope from the function header.

**`RestoreSpecific(procName, hwnd, *)` — line 869 area:**

Inside the restore loop after each successful `WinShow`, call:

```ahk
try HiddenState_Remove(hwnd)
catch as e
    LogRescue("HiddenState_Remove failed for hwnd=" hwnd ": " e.Message)
```

**`RestoreAllInGroup(procName, *)` — wherever it iterates `group.windows`:**

Same `HiddenState_Remove(hwnd)` call per restored window.

### Startup discovery — `RescueOrphanedWindows()`

Called from `Initialize()` AFTER Velopack-lifecycle args have been handled and AFTER `DetectHiddenWindows(true)` has been set, BEFORE the first tray-icon registration. Sequence:

1. `entries := HiddenState_Read()`. If empty, return immediately.
2. Build `survivors := []`. For each entry:
   - `if (!DllCall("IsWindow", "Ptr", entry.hwnd, "Int"))`: skip (window gone)
   - `try { currentPid := WinGetPID("ahk_id " entry.hwnd) }`: catch → skip
   - `if (currentPid != entry.pid)`: skip (PID changed — process restarted or recycled)
   - `try { currentPath := ProcessGetPath(currentPid) }`: catch → skip
   - `if (currentPath != entry.procPath)`: skip (different process with same PID — recycling)
   - `if (DllCall("IsWindowVisible", "Ptr", entry.hwnd, "Int"))`: skip (user restored it externally)
   - Otherwise: push to `survivors`.
3. Always rewrite `hidden.json` with `survivors` (drops stale entries from disk).
4. If `survivors.Length == 0`: return — no UI shown.
5. Otherwise: show modal rescue dialog (next section).

The validation is microseconds per entry — three syscalls and a string compare — so a worst-case file with dozens of stale entries clears in well under 100ms during startup.

## UI

Modal dialog Gui, theme-aware, owner-window-less (so it can take focus during startup before the main app tray is up).

### Layout

```
┌───────────────────────────────────────────────────────────────────────┐
│ Restore hidden windows                                          [_][X]│
├───────────────────────────────────────────────────────────────────────┤
│ Found 3 windows hidden by a previous session of minimize-to-tray.     │
│ Uncheck any you want to leave hidden.                                 │
│                                                                       │
│ ┌─────────────────────────────────────────────────────────────────┐   │
│ │     Process       Window title                       Hidden at  │   │
│ │ ☑   Code.exe      minimize-to-tray.ahk               14:32      │   │
│ │ ☑   chrome.exe    Anthropic Console - Google Chrome  14:32      │   │
│ │ ☑   notepad++     *new 1                             14:33      │   │
│ └─────────────────────────────────────────────────────────────────┘   │
│                                                                       │
│                [Restore Selected]  [Restore All]  [Cancel]            │
└───────────────────────────────────────────────────────────────────────┘
```

- ListView with `+Checked` style. Checkbox column auto-added.
- Three data columns: `Process`, `Window title`, `Hidden at`.
- `Hidden at` formatted as `HH:mm` (date dropped — same-day restarts dominate; if the user genuinely waited overnight before restarting, the time alone is still adequate context).
- Window minimum size: ~640 × 280. Resizable by user (Gui `+Resize`); columns auto-stretch.
- All rows pre-checked by default (`LV.Add("Check", ...)`).
- Default focused control: **Restore All** button (so Enter dismisses the most-common path).
- Dialog reuses theme state — if `themeState == "dark"`, apply the same dark palette as the About dialog. DWM dark-mode title bar via `DwmSetWindowAttribute(hwnd, 20, &TRUE, 4)`.

### Button behavior

- **Restore Selected** — iterate `LV.GetNext(0, "C")` checked rows; for each, call `WinShow("ahk_id " hwnd)` and `HiddenState_Remove(hwnd)`. Close dialog. Unchecked rows remain in `hidden.json` (the user will see them again next launch — they didn't actively dismiss them).
- **Restore All** — ignore checkbox state; restore every survivor row; clear `hidden.json` via `HiddenState_Clear()`. Close dialog.
- **Cancel** — close dialog; call `HiddenState_Clear()`. Orphaned windows stay hidden on the desktop. User explicitly opted out of recovery; no re-prompt on next launch.
- **X close (title-bar close button)** — same as **Cancel**.

### Per-window restore semantics

`WinShow("ahk_id " hwnd)` reveals the window in its prior position. The window:

- Reappears at its original screen coordinates.
- Reclaims its normal taskbar entry.
- Is NOT re-registered in the `Groups` map.
- Is NOT re-bound to any tray icon.

If `WinShow` fails (window died between validation and the actual restore action), log to `rescue.log` and continue with the remaining selections. No user-facing error — silent best-effort.

## Diagnostic hotkey — `Ctrl+Alt+Win+Z`

Bound globally (not `#HotIf`-gated). When pressed:

1. Capture the foreground window: `hwnd := WinGetID("A")`. If zero, show toast "No active window" and return.
2. Gather:
   - `hwnd` — hex + decimal
   - `class` — `WinGetClass("ahk_id " hwnd)`
   - `title` — `WinGetTitle("ahk_id " hwnd)`
   - `pid` — `WinGetPID("ahk_id " hwnd)`
   - `procName` — `ProcessGetName(pid)`
   - `procPath` — `ProcessGetPath(pid)`
   - `GWL_STYLE` — `DllCall("GetWindowLongW", "Ptr", hwnd, "Int", -16, "UInt")` formatted hex
   - `GWL_EXSTYLE` — `DllCall("GetWindowLongW", "Ptr", hwnd, "Int", -20, "UInt")` formatted hex
   - `GA_ROOT` — `DllCall("GetAncestor", "Ptr", hwnd, "UInt", 2, "Ptr")`
   - `GA_ROOTOWNER` — `DllCall("GetAncestor", "Ptr", hwnd, "UInt", 3, "Ptr")`
   - `DWMWA_CLOAKED` — read via `DwmGetWindowAttribute(hwnd, 14, &outBuf, 4)` — value 0/1/2/4
   - `DWMWA_SYSTEMBACKDROP_TYPE` — read via `DwmGetWindowAttribute(hwnd, 38, &outBuf, 4)` — value 0=auto / 1=none / 2=mica / 3=acrylic / 4=tabbed-mica
   - `IsWindowVisible` — `DllCall("IsWindowVisible", "Ptr", hwnd, "Int")`
   - `IsIconic` — `DllCall("IsIconic", "Ptr", hwnd, "Int")`
3. Format as multi-line text:

```
=== minimize-to-tray window dump ===
captured at:     2026-05-23T14:42:18Z
app version:     1.0.7
hwnd:            0x001E0A1C (1968412)
class:           Chrome_WidgetWin_1
title:           minimize-to-tray.ahk - minimize-to-tray - Visual Studio Code
pid:             13456
procName:        Code.exe
procPath:        C:\Users\jscha\AppData\Local\Programs\Microsoft VS Code\Code.exe
GWL_STYLE:       0x16CF0000
GWL_EXSTYLE:     0x00040100
GA_ROOT:         0x001E0A1C (same)
GA_ROOTOWNER:    0x001E0A1C (same)
DWMWA_CLOAKED:   0
SYSTEMBACKDROP:  2 (mica)
IsWindowVisible: 1
IsIconic:        0
```

4. `A_Clipboard := text`
5. Show toast: a small non-modal Gui with a single Text control, auto-dismiss after ~2.5s, in the bottom-right of the primary monitor. Reuse existing toast infra if any; otherwise add a small `ShowToast(message, durationMs := 2500)` helper.

The hotkey is intentionally documented INTERNALLY (no README mention) — it's a developer aid for v1.0.8 prep, not a user-facing feature.

## Failure modes

| Failure | Behavior |
|---|---|
| `hidden.json` corrupted (invalid JSON) | `HiddenState_Read` catches, returns `[]`, logs to `rescue.log`. App proceeds as if file was empty. Next hide rewrites the file fresh. |
| `hidden.json` write fails (disk full / readonly / permissions) | Logged; hide operation still succeeds (window is hidden, just untracked). Next-session rescue won't offer this window but the user can recover via the running app's tray icon. |
| `HIDDEN_STATE_TMP` orphaned from a crash mid-rename | Idempotent — next write overwrites it. No cleanup needed. |
| `WinShow` fails during rescue restore | Logged; remaining restores continue. User sees other windows return; the failed one stays hidden (probably already dead). |
| Rescue dialog opens but user kills the app via Task Manager before responding | `hidden.json` is unchanged (no Restore/Cancel applied). Next launch prompts again — desired behavior. |
| `LOCALAPPDATA` env-var missing or empty | `Initialize()` falls back to `A_AppData "\bilbospocketses\minimize-to-tray"`. If that also fails, rescue is disabled silently for this session. Hide/restore still works in-session. |
| Multiple AHK instances (shouldn't happen — `#SingleInstance Force`) | The second instance's `--veloapp-*` exit-fast path runs before any rescue; non-lifecycle launches replace the first instance. Single-writer guarantee preserved. |

## Testing plan

### Rescue mode — manual

1. Launch compiled `minimize-to-tray.exe`. Hide three different windows via `Win+Shift+Z`: e.g. Notepad, Chrome, Explorer.
2. Verify `hidden.json` contains three entries with correct procName / title / hiddenAt.
3. Force-kill the AHK process via Task Manager → End Task.
4. Relaunch `minimize-to-tray.exe`.
5. Expected: rescue dialog appears with three rows, all checked.
6. Click **Restore All**. Expected: all three windows reappear; `hidden.json` is `{"version":1,"windows":[]}`.

### Rescue mode — edge case: stale entries

1. Hide one Notepad window. Close Notepad via Task Manager (kills the process).
2. Force-kill AHK process.
3. Relaunch.
4. Expected: no rescue dialog (the only entry was pruned silently). `hidden.json` rewritten as empty.

### Rescue mode — edge case: Cancel

1. Hide one window. Force-kill AHK process. Relaunch.
2. Rescue dialog appears. Click **Cancel**.
3. Expected: dialog closes; `hidden.json` cleared; the hidden window is now an orphan on the desktop.
4. Relaunch again. Expected: no rescue dialog (bookkeeping cleared).

### Rescue mode — edge case: Restore Selected with mixed check state

1. Hide two windows. Force-kill AHK process. Relaunch.
2. Rescue dialog: uncheck row 1, leave row 2 checked. Click **Restore Selected**.
3. Expected: window 2 reappears, window 1 stays hidden. `hidden.json` retains only the entry for window 1.
4. Relaunch. Expected: rescue dialog with just window 1.

### Diagnostic hotkey

1. Focus a normal app (Notepad). Press `Ctrl+Alt+Win+Z`.
2. Paste from clipboard. Expected: structured dump matching the format above.
3. Focus VS Code. Press `Ctrl+Alt+Win+Z`.
4. Paste. Expected: structured dump with `Chrome_WidgetWin_1` class, mica/acrylic system-backdrop value, etc.
5. Share dump output back for v1.0.8 design.

### Regression

- Normal hide / restore via `Win+Shift+Z` still works (file writes do not block the hotkey).
- Middle-click title-bar minimize still works.
- About dialog opens normally, theme toggle still works, update dot still appears when applicable.
- Velopack lifecycle args still exit cleanly.
- `--veloapp-install` still seeds Run-on-login and Theme.
- First-run About dialog after fresh install still appears once and clears the marker.

## Open questions

None. All design decisions resolved in the 2026-05-23 brainstorming session:

- Trigger: auto-detect on startup only (no CLI, no tray-menu entry).
- Discovery: persistent bookkeeping only (no broad system scan).
- UX: modal list dialog (Restore Selected / Restore All / Cancel).
- Storage: filesystem JSON in `%LOCALAPPDATA%` (registry keeps scalar config only).
- JSON: hand-rolled parser / encoder (no vendored dep).
- v1.0.7 ships rescue + diagnostic only; actual Electron / Chromium fix moves to v1.0.8 after diagnostic output is shared.
